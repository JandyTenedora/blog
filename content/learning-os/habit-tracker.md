## The three habits

Three things get tracked every day: **German**, **Data Engineering**, and **Training**. Not outcomes — just whether the session happened. The assumption is that showing up consistently compounds, and that tracking streaks makes skipping feel more deliberate.

Each is given a daily time budget:

| Habit | Target | When |
|-------|--------|------|
| Data Engineering | ~10.5 hrs/week | Morning sessions |
| German | ~4.5 hrs/week | Evening or commute |
| Training | — | Separate from study time |

## How the tracking works

Every daily note starts with the same callout block:

```
> [!habits]+ Habits
> - [ ] German
> - [ ] Data Engineering
> - [ ] Training
```

The `obsidian-tracker` plugin scans the `Daily Notes/` folder for completed tasks matching those exact labels. Checking a box — `- [x] German` — registers as a completed habit for that day. The plugin renders streaks and completion rates in `Habit Tracker.canvas`.

The label text is load-bearing. Renaming "Data Engineering" to "Engineering" or reordering the list would silently break the streak data. The template is locked and the CLAUDE.md instructs agents never to touch it.

## Creating TODOs — the study plan

Each morning, `/dailystudy` reads the active learning roadmap, checks recent daily notes for context, and generates a study plan for the day. It writes the plan directly into the daily note under a `## Study Plan` heading:

```markdown
## Study Plan

- [ ] CS:APP Ch 3 — Machine-Level Representation of Programs (60 min)
- [ ] P1 HTTP Server — implement request parser (90 min)
- [ ] German — Hugo Week 3 recap + 10 new vocab (30 min)
```

Each task is a standard Obsidian checkbox. Tasks are scoped, timed, and tied to a specific milestone or reading chapter. The skill cross-references the roadmap to decide what's due — it doesn't invent tasks.

## Ticking things off — the study log

At the end of a session, `/dailylog` writes a structured log of what was actually done. It reads the study plan, asks what happened, and appends under `## Study Log`:

```markdown
## Study Log — 21:30

### Notes Touched

- [[CS:APP - Chapter 3 Notes]] — created, sections 3.1–3.4 covered

### Questions & Answers

> [!note] Question
> **Q:** What's the difference between %rsp and %rbp?
> **A:** %rsp is the stack pointer (tracks top of stack), %rbp is the base pointer (marks the current stack frame base). In modern compilers, %rbp is often omitted and offsets are calculated directly from %rsp.

### What I Learned Today

- x86-64 uses a 16-byte stack alignment requirement before a `call` instruction
- `leaq` is frequently misused for arithmetic — it doesn't actually load anything from memory
```

Completed plan tasks get checked off. If a task wasn't done, it stays unchecked — no false completion. The log becomes a searchable record of what was understood, not just what was attempted.

## The daily note structure

A complete daily note looks like this:

```
> [!habits]+ Habits
> - [x] German
> - [x] Data Engineering
> - [ ] Training

---

## Study Plan

- [x] CS:APP Ch 3 — Machine-Level Representation (60 min)
- [ ] P1 HTTP Server — implement request parser (90 min)

---

## Study Log — 21:30

[generated by /dailylog]
```

The habits callout sits at the top so the tracker plugin finds it reliably. The study plan and log follow. The separation is deliberate — the plan is a commitment, the log is a record of reality.

## Visualisation

`Habit Tracker.canvas` renders streak data from the `obsidian-tracker` plugin. It shows completion rates per habit over rolling windows. Useful for noticing drift before it becomes a gap — a week where German drops to two sessions is visible before it becomes three weeks.
